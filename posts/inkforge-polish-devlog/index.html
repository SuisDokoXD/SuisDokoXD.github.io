InkForge 打磨记：从「能用」到「顺手」的一次 UI/UX 收口

发表于2026-05-30|更新于2026-05-30|项目记录
|总字数:2.1k|阅读时长:6分钟|浏览量:

上一篇《InkForge 开发实录》把引擎、数据、IPC 全景拆了一遍。这一篇相反——没有任何新功能，全是”看着难受所以动手改”的细节打磨。功能闭环之后，决定一个工具用着舒不舒服的，往往就是这些不起眼的地方。

0. 为什么做这一轮

beta.12 之后，InkForge 该有的都有了。但真正每天用，几个地方一直硌得慌：

  • 左侧导航栏一排彩色 emoji，和干净的蓝灰主界面凑在一起，像两个设计师打架；
  • 切到白色主题，顶部标题栏还是黑的；
  • 新建项目第一眼，编辑区一行灰字飘在大片空白里，不知道下一步干嘛；
  • 桌宠是纯手写 SVG 画的，怎么看怎么丑；
  • 窗口一拖拽缩放就卡顿掉帧。

这次就专门收这些口。

1. 图标系统：彩色 emoji → lucide 单色

最显眼的”不协调”来源：ActivityBar 用 emoji 当导航图标（✍🧩👥🎭🌍…）。emoji 由系统字体渲染，颜色、风格、基线全不可控，和系统蓝 + 灰阶的整体调性完全打架。

换成 lucide-react 单色线性图标，13 个主视图各配一个语义图标：

统一 strokeWidth={1.75}、18px，选中态走 accent 蓝高亮。底部的命令面板、设置也一并换成 Command / Settings 图标。一排单色线条下来，整个侧栏瞬间安静、协调。

2. 配色收口：amber → 系统蓝 accent

项目早期残留了一套 amber-*（琥珀橙）色值，和后来确立的系统蓝 accent-* 体系并存——有些按钮是橙的，有些是蓝的，毫无规律。

直接做了一次全量 codemod：把 amber- 全部替换成 accent-，覆盖 90 个文件、337 处引用，再删掉 tailwind.config.cjs 里已经没人用的 amber 别名块。配色从此只有一套蓝。

教训：颜色别名要么从第一天就定死，要么趁早 codemod。拖到几十个文件都在用了，每多一天都更难收。

3. 主题适配：白色模式不该是黑的

TitleBar 之前把深色渐变背景写死成 hex，白色主题下自然还是黑的；上面的高光线、状态胶囊、窗口按钮 hover 也都是”白字配深底”的硬编码。

改法是让每一处都带 dark: 变体，浅色走浅、深色走深：

顺手把 BrandMark logo 也重画了：原来那版在白底上带一圈琥珀辉光 + 深色叠层，糊成一团。新版是系统蓝 squircle + 白色羽毛笔，浅深底都干净。窗口底色也从深色 #0a0e1a 改成浅色 #f7fafe，缩放时新露出的边缘不再闪一帧深色。

4. 空状态：从”一行灰字”到”有指引、有落点”

最影响第一印象的，是各页面的空状态。比如新建项目后进写作页，整个编辑区只有一行：

大片空白里一行小灰字，既不知道点哪，也显得页面”没做完”。

把这些 bare 文案统一换成项目里早就有、却没用起来的 EmptyState 组件——大图标 + 标题 + 说明 + 可点的主按钮：

  • 写作页：✍ +「开始你的第一章」+ 一个直接能点的 新建本章 按钮（回调从 WorkspacePage 透传进来）；
  • 人物页：👤 + 选择/导入角色的引导；
  • 来信页：✉ + 选信查看的引导。

空状态从”系统好像卡住了”变成”我知道下一步该干嘛”。其余页面那些已经是带边框引导卡的空状态（如各页”未选择项目”）就没动——没必要为了统一而统一，强行改版只会引入看不见的回归。

5. 桌宠重构：4 物种手绘 SVG → 单一 Lottie 吉祥物

桌宠原来用 600 多行纯手写 SVG 拼了 4 个物种（猫 / 狐 / 鸮 / 章鱼），靠代码摆形状，可爱上限极低。

这次方向定为：收敛成 1 个吉祥物，引入 Lottie 动画（接受 Lottie Simple License，可商用免署名，分发随附许可文本）。

技术上用 lottie-react：

新建 CompanionLottie 顶替原来的 PetSprite（整文件删除），props 对齐。一个关键取舍要讲清楚：

现成的 Lottie 没法复刻原来 11 种状态的专属造型（番茄钟戴眼镜、dizzy 转圈等）。所以分工是：Lottie 只负责基础角色 + idle/typing 的可爱循环，状态表达继续靠与物种无关的手段——容器级 CSS 关键帧（bob/jump/breathe）、已有的粒子 / 节日 / 番茄钟环叠层，以及按状态微调 Lottie 播放速度（typing 略快、sleepy 放慢、cheering 加速）。

代价是状态表现少几分”专属感”，换来的是角色本身可爱得多。companion-store 里的 pet / secondaryPet 多物种字段、persona 的物种分支也一并收敛成单一吉祥物（默认名「墨墨」）。reduced-motion 下 Lottie 暂停为静态首帧，尊重系统的”减弱动态效果”。

集成层我搭好了——换文件即生效，最终那个”软萌”角色由我自己挑素材替换 mascot.json 即可，不用改代码。

6. 窗口缩放流畅度：25 层模糊的代价

拖拽缩放窗口明显卡顿。排查下来，根因是界面里铺了二十多层 backdrop-filter: blur（玻璃质感的卡片、弹层、标题栏）。静止时没问题，但 resize 期间每一帧都要重新计算这些模糊区域，重绘成本爆炸。

解决思路：只在缩放进行中临时关掉模糊，静止时立刻恢复。

写了个 use-window-resize-perf hook——监听 resize，给 <html> 挂上 win-resizing 类（rAF 触发，160ms debounce 移除），CSS 里：

拖拽时模糊全停，掉帧消失；松手 160ms 后玻璃质感回来，肉眼几乎察觉不到切换。SelectionToolbar 的 scroll/resize 重算也顺手用 rAF 合并了。

7. 页面切换 & 引导页动效

主视图切换原来是瞬间硬切。包了一层 AnimatedPage（基于 motion 的 AnimatePresence mode="wait"），13 个页面切换时淡入 + 轻微上移，贴近 macOS 单窗口切换的清晰感；引导页步骤之间也加了过渡。所有动效都接 useReducedMotion，关了系统动画就退化成纯淡入淡出，不强加位移。

8. 顺手做的工程化

  • 删掉死代码：没人用的 useAppShortcuts 钩子、amber Tailwind 别名；
  • 新增 verify-skill.cjs：skill 仓库往返 + skill-engine 集成冒烟测试，挂进 verify:all；
  • 新增 swap-sqlite-abi.cjs：一键诊断 / 切换 better-sqlite3 的 Node↔Electron ABI（上一篇吐槽过的双 ABI 噩梦，这次给了个趁手的工具）。

9. 这一轮的数字

新增依赖 lottie-react

写在最后

这一轮没有任何一行是”新功能”，全是”原来能用，但用着难受”的地方。功能做到闭环之后，真正拉开体验差距的就是这些细节：图标统不统一、空状态有没有指引、缩放卡不卡、白天模式是不是真的白。

设计哲学没变——本地优先、数据是用户的。只是这次多补一句：工具不光要能用，还得让人愿意一直用下去。

如果你也在做 Electron 桌面应用，这几个点（backdrop-blur 缩放性能、主题硬编码、空状态引导）大概率你也会踩到，欢迎在 GitHub Issues 交流。
