InkForge M10 继续收口：把 beta.19 后的 WIP 逐项落地

发表于2026-06-19|更新于2026-06-19|项目记录
|总字数:2.5k|阅读时长:7分钟|浏览量:

上一篇《InkForge 进度同步：从 beta.12 到 beta.19，先把写作链路打稳》写到 beta.19，并提到 6 月 18-19 日还有一批 M10 polish WIP 没完全收束。今天这篇就是把那部分继续往前推后的记录：不急着换版本号，先把已经暴露出来的隐患一个个按住。

这次的关键词还是 M10：体验打磨、工程边界、数据可信、发布可靠性。

0. 这次补上了什么

方向 进展
IPC 边界 补齐 validation 测试，覆盖 settings、provider、review、world-pack、character、tavern 等高风险入口
World Pack 融合预览保存时不再二次调用模型，直接保存用户审核过的预览内容
写作输入 编辑器、章节缓存、世界观详情页继续加固，避免旧数据回流覆盖新输入
状态反馈 多处短状态提示改成可清理计时器，组件关闭或重开时不会被旧 timer 反向改状态
UI 动效 继续替换零散 Tailwind 动画类，统一走 motion token / MotionSpinner，并照顾 reduced motion
验证 桌面 typecheck、根 typecheck、desktop build、desktop test、a11y 静态扫描、diff check 均通过过

上一篇已经讲过“为什么要把大纲页的正文生成职责收回 AI 写作页”。这一篇更偏工程收尾：功能看起来可能不大，但它们决定软件在真实使用中会不会突然让人失去信任。

1. IPC validation 从“改了”变成“有测试守住”

Electron 的 renderer 到 main 之间，TypeScript 只能在开发时帮忙；真正运行时传进来的还是普通对象。上一轮已经把大量 ipcMain.handle(... input: XxxInput) 的入口改成 runtime parse，这一轮继续补回归测试。

现在 validation 测试覆盖了几类最容易出事的边界：

  • settings:set 只允许已知设置项，并按每个 key 的真实类型解析。
  • provider / model 相关字段走白名单，不让任意字符串直接落到主进程逻辑里。
  • review、world-pack、character、tavern 这些会读写内容或触发模型调用的入口，都有输入结构校验。
  • 静态扫描继续防止以后又把 IPC handler 写回裸 typed input 参数。

这类测试不会让界面多一个按钮，但它会减少“以后改着改着又绕过解析层”的概率。

我越来越觉得桌面 AI 工具里，IPC validation 不应该被当成安全附加项。只要主进程能碰文件、数据库、模型配置、导入导出，它就是产品可信度的一部分。

2. World Pack 融合：保存的是用户看过的那一版

这次还修了一个语义上很重要的问题：World Pack 融合预览。

之前的流程里，用户先生成融合预览，满意后再点“保存为新卡”。如果保存动作内部重新调用一次模型，那就会出现一个危险错位：

用户审核的是 A，最后落库的可能是 B。

这对写作设定工具很致命。世界观卡不是临时推荐语，它会进入后续写作参考。如果用户已经看过并确认某一版内容，保存就应该保存这一版，而不是再让模型自由发挥一次。

现在保存语义被收紧：

  • 预览阶段生成 suggestion。
  • 用户点击保存时，把这份 suggestion 作为输入传回主进程。
  • 主进程 validation 校验这份 reviewed suggestion 的结构。
  • service 直接把 reviewed suggestion 持久化。
  • 插入 pack 和 entries 放在同一笔事务里完成。

这样之后，World Pack 融合更像一个可审核的创作工具，而不是“看一版、存另一版”的黑盒。

3. 继续处理“输入后消失，刷新后又出现”

上一篇提到过这个问题：输入文字后内容像是消失了，但刷新后又出现。这个现象很迷惑，因为它不是单纯“保存失败”，而是保存成功后，旧数据又回流覆盖了本地新状态。

这次继续把防线补细：

  • NovelEditor 跳过编辑器自己刚发出的 onChange 回流，避免 value 同步又把正在输入的内容 setContent 回旧版本。
  • 中文 IME 合成过程中不做外部 setContent，减少输入法过程里的误覆盖。
  • EditorPane 保存成功、快照恢复后同步更新 chapter-content 查询缓存，不等父级刷新拿旧正文回来。
  • 世界观详情页用 loaded entry id、dirty snapshot 和 updatedAt 判断外部数据是不是过期，不轻易覆盖正在编辑的条目。

写作软件的底线不是“功能多”，而是“用户打进去的字不能莫名其妙丢”。哪怕最后发现数据库里还在，只要界面上突然消失，用户都会先怀疑这个工具不可靠。

这类问题不适合靠“刷新一下就好了”来解释。程序应该主动分清楚：

  • 这是用户刚输入的本地状态；
  • 这是刚保存成功的内容；
  • 这是查询缓存里上一次的旧内容；
  • 这是父级刷新带回来的外部数据。

只有这几层不互相踩，编辑体验才会稳定。

4. 状态计时器不能散落在各处

M10 里很大一部分改动都跟“短状态反馈”有关。

很多界面会有这样的代码：成功后显示“已保存”，然后 setTimeout 两秒后消失。单独看没问题，组件多起来就会出事：

  • 弹窗关了，timer 还在后台跑。
  • 用户马上重新打开，旧 timer 把新状态清掉。
  • 失败信息还没看清就被成功状态覆盖。
  • 组件卸载后继续 set state，虽然不一定立刻崩，但行为很脏。

这次继续收了几条典型路径：

  • WorldPackEditDialog 的保存状态从裸 setTimeout 改成可清理的 timer helper。
  • 封面上传开始时进入保存中，成功后短暂显示已保存，失败回到空闲。
  • PackSlotPanel 复用 useTimedStatus，移除、启停参考、调整顺序都有短暂成功提示；错误保留，并用 role="alert"。
  • ReviewReportPanel 关闭修复预览后延迟清空内容的 timer 可以取消，避免旧关闭动画清掉新预览。
  • 桌宠的临时状态恢复、每日达成延迟标记、双击和长按 timer 都在禁用或卸载时清理。

这些都不是大功能，但它们会让软件少很多“刚才明明点了，怎么状态又变了”的错觉。

5. 动效继续统一，但不追求花

之前很多地方直接用了 animate-spin、animate-pulse、animate-bounce、transition-all。这些类方便，但在一个大型桌面应用里会逐渐失控：每个组件转法不一样、禁用动效不好处理、按钮宽度容易抖、状态切换也不稳定。

现在继续往统一方向收：

  • loading 用 MotionSpinner / MotionSpin。
  • 进入、退出、hover、tap 尽量走 motion-tokens。
  • reduced motion 下少动或不动。
  • 用稳定尺寸和明确 disabled state 减少布局跳动。
  • icon-only 按钮补 aria-label / title。

这个阶段的目标不是让 InkForge “更炫”，而是让它更安静、更稳定、更像一个可以长时间使用的写作工作台。

6. 当前验证状态

这轮改动在本地跑过多组检查。比较关键的结果：

  • pnpm --filter @inkforge/desktop typecheck：通过。
  • pnpm typecheck：通过。
  • pnpm --filter @inkforge/desktop build：通过。
  • pnpm --filter @inkforge/desktop test：通过，桌面端单测覆盖了新增 validation 和世界图逻辑等路径。
  • 静态可访问性扫描：没有报告具体问题。
  • git diff --check：通过。
  • renderer 旧动效类扫描：没有发现本轮关注的 animate-spin / transition-all 回退。
  • dev server 错误日志为空，5173 本地开发服务正常。

另外，SQLite 当前在 Electron ABI，适合跑本地桌面开发版；如果要跑某些 Node 侧 verify 脚本，需要先切回 Node ABI。这点已经记录成项目经验，避免把 native addon ABI 问题误判成 TypeScript 回归。

7. 还没有结束

M10 还不是“所有 polish 都完成”。现在本地工作区仍然有大量改动，后面更适合按主题拆分：

  • IPC validation 模块继续补更细的边界单测。
  • EditorPane / SkillPage 这类巨型 UI 文件可以继续拆，但需要更严格的视觉回归检查。
  • World Graph 还可以补图例、批量关系操作、结构摘要导出，以及和 AI 写作上下文联动。
  • World Pack 融合、角色卡、酒馆、审稿这些高频路径还可以继续统一状态反馈和错误恢复。
  • 0.1.x Windows 构建仍未签名，正式分发前还需要解决代码签名。

但至少这一轮之后，上一篇里提到的“还有一批本地 WIP”已经有一部分从 WIP 变成了可验证的改动。

写在最后

InkForge 现在最耗时间的已经不是“把一个功能做出来”，而是把功能之间的边界、状态、反馈、保存语义和验证链路逐渐磨平。

AI 写作工具尤其不能只追求生成能力。真正长期使用时，用户更在意的是：

  • 我写下的字会不会丢；
  • 我确认过的设定会不会被偷偷换掉；
  • 按钮点下去之后软件到底有没有在做事；
  • 错误发生时我能不能知道原因和下一步；
  • 本地数据是不是还在我自己手里。

所以这一轮 M10 还是那句话：不急着宣布大版本，先把真实使用里会刺人的地方继续拔掉。
