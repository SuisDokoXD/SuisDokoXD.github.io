InkForge 开发实录：一个本地优先 AI 小说写作工作台的全景拆解

发表于2026-05-26|更新于2026-05-26|项目记录
|总字数:3.4k|阅读时长:13分钟|浏览量:

之前发过一版基于 commit history 的总结，回看发现严重低估了项目规模 —— commit message 太省，体现不出实际的代码量。这一版直接读代码重写，更接近项目的真实模样。

0. 一个数字开头的项目体检

维度 实际规模
Monorepo 项目 10 个（1 个 app + 9 个 package）
数据库 migration v1 → v26
数据库表 46 张
IPC channel 221 个
Main 进程 service 50+ 个
Storage repository 37 个
Renderer 页面 13 个
组件目录 / 单文件组件 17 个目录 + 29 个顶级组件
Zustand store 3 个（app / bookshelf / companion）
i18n zh / en / ja

这不是一周末敲出来的小玩具 —— 这是一个功能闭环的中等规模 Electron 应用。

1. 项目是什么

InkForge 是一个本地优先的 AI 小说写作桌面工作台。三句话讲清楚：

  1. 所有数据存本地。章节正文进磁盘 .md 文件，元数据进 SQLite，AI provider 自带 key —— 没有云、没有服务器、没有遥测。
  2. 不只是编辑器。围绕”写小说”这件事，集成了大纲、世界设定、角色卡、4-Agent 自动写作、审稿、素材库、联网研究、命令面板、桌宠陪伴。
  3. 可编程。Skill 模板系统让用户能自定义 AI 行为，不需要每次写 prompt。

GitHub: tompignofind123-cyber/inkforge

2. 技术栈与”为什么”

选型理由（每条都是踩过坑后的决定）：

  • Electron 32：跨三端一次打包。代价是包大、Node 原生模块 ABI 要小心 —— 这成了项目早期最痛的事。
  • better-sqlite3：同步 API，本地场景无需 await 满天飞。但它是 native module —— 见后文 “踩过的坑” 第 1 条。
  • TipTap 2：基于 ProseMirror，长文本性能好，扩展模型清晰。
  • Zustand：3 个 store 分领域（全局 app / 书架 / 桌宠），不用 Redux 那套 boilerplate。
  • React-Query：所有 IPC 调用的远端状态都是 query，缓存、失效、refetch 全交给它。
  • cmdk：命令面板。世界书 / 章节 / 技能切换全靠它。
  • @xyflow/react：世界书的关系图可视化。
  • xterm.js + node-pty：UI 里内嵌一个真终端，用来跑 Skill 脚本和看输出。
  • electron-updater：自动更新。

3. Monorepo 真实结构

构建依赖链：shared → storage → 其它 packages → desktop。

数据流

流式调用走 async iterator + BrowserWindow.webContents.send 推 chunk 给 renderer，事件有 phase / chunk / snapshot / done 四种。

4. 五大引擎深挖

4.1 AutoWriter Engine —— 4-Agent 流水线

整章自动生成的核心。流水线如下：

关键设计：

  • OOC-gate：检测段落是否”出戏”（character 行为是否偏离 character card 设定），与 Critic 并列工作
  • Per-role binding：4 个 agent 可以分别绑定不同的 provider/model/temperature。比如 Planner 用 Claude Opus（推理强），Writer 用 GPT-4o（中文文笔好），Critic 用本地 deepseek-r1，Reflector 再换一个 —— 一次写作可以混用 4 个模型
  • 事件流：4 phase 之间通过 streaming event 反馈进度，UI 可以实时看到当前 agent 在工作
  • 跨章节记忆：Reflector 的产物写入 chapter_summaries 表（v21 加的），下一章启动时 AutoWriter 会拼接全部前序章摘要 + 最近 1 章细节注入 prompt，解决长篇 OOC / 世界观漂移

之前那版我把第 4 个 agent 写成 Editor，错了 —— 实际上是 Reflector（提纲师 / 写手 / 审稿员 / 反思员），canonical 类型见 packages/shared/src/domain.ts AutoWriterAgentRole。这次专门读了代码核对。

4.2 Skill Engine —— 自研 macro 模板

之前说这是”类 Jinja”的模板语言。又错了 —— 实际上是借鉴 SillyTavern macros 的自研系统：

配合 world-info-activator，根据 world entry 的关键词、selectiveLogic（and_any / not_all / not_any / and_all）、命中概率，自动决定哪些设定条目要注入到 prompt 上下文。激活轨迹会写入 world_info_trace 表，可在 WorldInfoDiagnosticPanel 里复盘。

4.3 Tavern Engine —— 多角色对话

不只是简单的 chat —— 这是一个有 director 的多角色调度系统：

  • 三种 role：director（导演角色，决定下一句谁说）、character（被扮演的角色）、summary（摘要压缩 agent）
  • 两种模式：director 模式（导演显式指挥）、auto 模式（角色自由轮转）
  • Token budget tracking：监控上下文用量，超限触发 summarizer 压缩历史
  • Last-k 滑窗：保留最近 k 条原文，更早的走 summary
  • character_sync_log：人物状态变化（情绪 / 关系 / 知识）跨场景同步

4.4 Review Engine —— 5 维度审稿

维度 检查什么
人物一致性 角色行为 / 语气是否前后矛盾
时间线 事件先后是否合理，是否有时间穿越
伏笔回收 前文埋的钩子有没有被回收
世界观 设定是否被违反
语言风格 文风是否统一

每个 finding 三级严重度：info / warn / error，带 excerpt 高亮原文位置。还可以基于 Skill 自定义维度 —— 用户可以加自己的审稿规则。

4.5 Research Core —— 联网研究

三种搜索源任选：Tavily / Bing / SerpAPI，外加 LLM fallback。统一抽象成 ResearchProvider 接口，返回标准化的 ResearchSearchHit[]。结果可以保存为 research_notes，供后续 AI 生成调用。

5. 数据层 —— 26 个 migration 的演进史

packages/storage/src/migrations.ts 是项目历史的最佳化石记录。挑几个关键节点：

版本 加了什么 意味着什么
v1 projects / chapters / providers / ai_feedbacks 项目骨架
v2 outline_cards / daily_logs / app_settings 大纲卡 + 写作打卡
v3-v4 skills / tavern_cards / characters Skill 系统、Tavern 雏形
v5-v6 character_sync_log / tavern_sessions / tavern_messages 多角色对话完整化
v8-v10 world_entries / research_notes / review_dimensions/reports/findings 世界书 + 研究 + 审稿三大体系
v12 provider_keys（多 key 策略） 一个 provider 多 key 轮转
v13-v18 book_cover / chapter_log / chapter_snapshot / auto_writer_runs / achievements / character_letter / scene_binding / sample_lib / rag / world_relationship 一波”书房”特色功能集体落地
v19-v20 projects 表补 synopsis / genre / subGenre / tags / master_outline + global_worldview 早期 schema 假设过简，被迫追加
v21-v26 chapter_summary / world_pack / author_note / character_card_import / voice_profile / world_info_trace 卡牌化重构 + 上下文注入三件套

46 张表全景（按域分组）：

  • 核心：projects, chapters, providers, ai_feedbacks
  • 大纲与进度：outline_cards, daily_logs, app_settings
  • LLM 与技能：skills, provider_keys
  • 角色与对话：tavern_cards, characters, character_sync_log, tavern_sessions, tavern_messages, character_card_import, voice_profile
  • 世界观：world_entries, world_pack, world_pack_entries, project_world_pack_slots, world_relationships, world_info_trace, global_worldview
  • 审稿：review_dimensions, review_reports, review_findings
  • 场景绑定：scene_binding
  • 素材与 RAG：sample_lib, sample_chunks
  • 书架与元数据：book_cover, chapter_log, chapter_log_entries, chapter_origin_tag, chapter_snapshot, auto_writer_run, achievements, character_letter, chapter_summary, author_note
  • 研究：research_notes

关于 RAG 的一个澄清

之前我说 InkForge 有”RAG 向量检索”。这个表述不准确 —— 读了 rag-repo.ts 后真相是：

当前实现是基于 SQL LIKE 的关键词模糊匹配，不是向量嵌入。智能之处在 rag-smart-router.ts 里：对中文做 2-char window tokenization，配合 world_entries / characters / research_notes 三个来源按场景路由。

这不是 RAG 的标准定义，但它够用 —— 本地小说写作场景，关键词召回 + LLM 做最终筛选，远比拉一个 embedding 模型实在。未来如果做 vector DB，hook 是留好的。

6. UI 层 —— 13 个页面 + 桌宠 + 内嵌终端

13 个页面

页面 干什么
WorkspacePage 主写作面板（编辑器 + 章节树 + AI 工具栏）
OutlinePage 大纲卡管理
AutoWriterPage 4-Agent 自动写作配置 + 运行
CharacterPage 小说人物管理
SkillPage Skill 管理 + 编辑器
TavernPage 多角色对话室
WorldPage 世界书 + 关系图 + 命令面板
ResearchPage 联网研究 + 笔记
ReviewPage 审稿报告
MaterialsPage 素材库
LetterInboxPage 角色来信收件箱
AchievementHallPage 成就殿堂
OnboardingPage 新项目引导

桌宠系统（companion-store）

不是花瓶 —— 是一套完整的子系统：

  • Companion.tsx：本体
  • CompanionBubble.tsx：对话气泡（”被点击时”/“长时间没动”/“达成目标”各有反应）
  • CompanionChat.tsx：跟桌宠对话
  • CompanionParticles.tsx：粒子特效
  • CompanionPomodoroRing.tsx：番茄钟环
  • CompanionFestiveOverlay.tsx：节日特效（春节、生日等）

桌宠状态由 zustand companion-store 管，有 petted 状态（被点击的反应）和 festive 模式（节日切换）。

内嵌终端

@xterm/xterm + @xterm/addon-fit + node-pty —— UI 里真的有一个 PTY 终端，可以跑命令、看 Skill 执行输出、做轻量脚本调试。不是”假的 console”，是真 shell。

场景绑定（Scene Binding）

一个用户友好的设计点：把”AI 在哪个场景下该用什么 provider”显式化。

  • 基础 5 场景：outline 生成 / 主体生成 / 抽取 / 总结 / 内联
  • 高级 9 场景：analyze / quick / chat / skill / tavern / auto-writer / review / daily-summary / letter

用户在 SceneRoutingPanel 里把场景 ↔ provider 拖拽绑定，不同任务自动路由到合适的模型。

7. Provider 系统 —— 不只是 “key 一个”

provider_keys 表（v12 加的）支持：

  • 多 key：同一个 provider（比如 OpenAI）配多个 key
  • 权重：不同 key 调用比例不同
  • 失败计数：达到阈值自动禁用
  • 冷却时间（cooldown_ms）：达到 rate-limit 后冷却

加上 v25 的 character_card_import 指纹追溯、v22+ 的 world info selectiveLogic 复杂选择 —— 这个项目里有大量你不细看代码看不出来的工程细节。

8. 踩过的坑

1. better-sqlite3 双 ABI 噩梦

Runtime Node 版本 ABI
系统 Node（跑 verify script） 22 127
Electron 32 主进程 内嵌 Node 128

同一份 prebuilt 喂不了俩 runtime。解决：

2. macOS 构建死循环

GitHub Actions 上 macOS runner 跑 build：

  • npm 10 已经移除 python 配置 → 删掉 npm config set python 步骤
  • electron-builder 卡在 DMG 生成 → 直接跳过 DMG，只出 zip
  • Python 3.11 + setuptools 必装，否则 native module 编译挂

3. ProjectRecord 太瘦

早期 ProjectRecord 只有 {id, name, path, createdAt, dailyGoal, lastOpened}。
后续移植 ainovel 时假设有 title / synopsis / genre / tags / masterOutline —— 崩了。
v19/v20 紧急 ALTER TABLE 补字段。

教训：核心 record 早期就该想清楚业务字段，不要等”用到再说”。

4. shared / storage 必须先 build dist

它们以 workspace package 的 dist 形式被消费，不是 src。typecheck desktop 之前必须先 build shared + storage，否则类型错乱。

5. Migration 永远 append-only

26 个 migration 全部是 append-only 的对象。永远不要改老版本，只加新对象。否则用户升级时数据库状态不一致，灾难。

6. IPC 走 TypeScript declaration merging

新增 IPC channel 不用动总入口，类型自动合并。这是大型 monorepo 的福音。

9. 让我重新认识项目的几件事

读完一圈代码，几个让我意外的发现：

  1. 不是 RAG，是 LIKE 关键词 + 智能路由。务实，够用，hook 留好将来切 vector DB。
  2. Skill 模板是自研 macro，灵感来自 SillyTavern。不重新发明轮子但也不硬塞 Jinja。
  3. AutoWriter 的 4 个 agent 可以混用 4 个 provider。Planner 用强推理模型，Writer 用文笔好的，Critic 用便宜的本地模型 —— 设计上就为”性价比”留了空间。
  4. 桌宠有节日特效。companion-festivals.ts 暗示春节会有特殊视觉反应，这种细节体现项目温度。
  5. UI 里真有终端。xterm + node-pty 不是装饰，可以真的跑命令。
  6. Scene Binding 是 14 个场景，不是简单的”chat / completion 两个分支”。
  7. provider_keys 是完整的密钥池设计，不是简单存一个 token。

10. 下一步

beta.12 之后想推的方向：

  • 真正的 vector RAG：把 LIKE 路径升级成 embedding（用 transformers.js 或本地 ollama 跑 nomic-embed）
  • 离线 LLM：接 llama.cpp 或 ollama，让全流程脱网
  • 章节版本对比 UI：snapshot 数据已经在了，缺一个 diff viewer
  • Skill 市场：素材 / Skill / 世界包的社区分享（market.ts IPC 已经先占位）
  • i18n 完善：英文 / 日文的 prompt 模板 + UI 翻译质量提升
  • Conventional Commits 强制：下版本提交规范化，让 commit history 自己能讲故事

写在最后

如果只看 30 条 commit message，会以为这是一个”小工具”。
但实际拆开 —— 10 个项目（1 app + 9 package）、46 张表、221 个 IPC、4-Agent 写作管线、3 种联网搜索源、5 维度审稿、可编程 Skill、桌宠系统、内嵌终端、关系图可视化、26 步 migration 演进 —— 这是一个完整的写作工作台。

设计哲学还是那句：“用户的数据是用户的”。本地优先、文件可读、provider 自选、不绑定云、不上传任何东西。希望能成为认真写小说的人手里趁手的工具。

如果你也在做类似项目，或对其中某个引擎实现感兴趣，欢迎在 GitHub Issues 找我聊。
