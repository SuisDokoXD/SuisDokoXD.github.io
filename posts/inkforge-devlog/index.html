InkForge 开发实录：30 天造一个本地优先的 AI 小说写作工具

发表于2026-05-26|更新于2026-05-26|项目记录
|总字数:1.7k|阅读时长:6分钟|浏览量:

项目是什么

InkForge 是一个本地优先的 AI 小说写作桌面应用。所有数据存在本地，不联网不上传 —— 章节正文走磁盘 .md 文件，元数据走 SQLite，AI provider 自带 key 自选服务商。

一句话：TipTap 编辑器 + 多 LLM 调度 + 本地数据库 + Electron 壳 拼成的写作工作台。

GitHub: tompignofind123-cyber/inkforge

技术栈与架构决策

为什么这么选：

  • Electron 32：跨三端 (win/mac/linux) 一次打包，原生菜单和文件 IO 都不用妥协。代价是包大、Node 原生模块 ABI 要小心。
  • better-sqlite3：同步 API，写代码不用 await 满天飞。本地写作场景没有高并发，同步就够。
  • TipTap：基于 ProseMirror，扩展模型清晰，长文本性能好。比 Slate 文档全、比 Lexical 生态成熟。
  • Monorepo (pnpm workspace)：业务拆 9 个 package，编辑器/存储/LLM/AI Agent 都能独立演进。

包结构

构建依赖：shared → storage → others → desktop。

数据流

30 天时间线

Week 1：搭骨架 + CI/CD 苦战（4/27 ~ 4/30）

日期 干了啥
4/27 首 commit，项目骨架
4/28 一天连发 13 个 commit 修构建工作流：pnpm 版本、Node 24、Electron 原生重建、release 流程
4/30 macOS 跑不通 —— 装 Python 3.11 + setuptools，跳过 DMG 避免 hang，再删掉 npm 10 已经移除的 python 配置项

Week 1 的核心痛点：better-sqlite3 native binding ABI 不匹配。Node 22 是 ABI 127、Electron 32 是 ABI 128，同一个 prebuilt 喂不了俩 runtime。最后靠 prebuild-install --runtime=electron|node 在不同场景切换。

Week 2：仓库爆炸期（5/2 ~ 5/4）

  • 5/2：一口气加了 7 个 repository：成就、AutoWriter Run、书籍封面、章节日志、章节起源标签、章节快照、角色来信
  • 5/3：更新 README，正式把书房 / AutoWriter / 桌宠 / 成就 / 角色来信几个特色功能纳入设计
  • 5/4：移植 ainovel 的核心功能 —— RAG / 大纲 / 世界书 / 样本库 / AI 生成。这是单 commit 改动最大的一笔，奠定了后续两周所有功能

踩到的坑：移植 ainovel 时假设 ProjectRecord 自带 title/synopsis/genre/summary/tags，结果实际只有 {id, name, path, createdAt, dailyGoal, lastOpened} 六个字段。verify-script 直接挂掉。后来全部改成 需要这些字段的功能各自加迁移，不再假设公共字段存在。

Week 3：UI 与体验打磨（5/6 ~ 5/16）

  • 5/6：素材管理 CRUD + UI 组件
  • 5/7：UI 大改 + AutoWriter 优化 —— 整个写作面板重做，AI 自动写作流程更顺
  • 5/16：修了一个隐蔽 bug —— 写作段落重新加载会吞空格。同一天另一个 commit 修了无法编译

Week 4：卡牌化重构（5/26）

最后一发巨型 commit：60 文件 / +8953 行。这一次重做了：

  1. 世界书包 (World Pack)：把世界条目重组成卡牌，带封面、命令面板、融合对话框
  2. 角色卡 CCv3：实现 CCv3 编解码器 + PNG 元数据读取，可以直接导入业界通用的角色卡 PNG
  3. 作者笔记 (Author Note)：独立的笔记面板和上下文注入通道
  4. 声音档案 (Voice Profile)：管理叙述声音/风格档案
  5. 世界信息激活器：在 skill-engine 加智能激活器 + 追溯仓库 + 诊断面板
  6. RAG 智能路由：按场景路由检索来源
  7. 审稿服务独立：拆出 review-service 与 review-finding-repo
  8. 融合提示词管线：build-fusion-prompt + parse-fusion-output，世界包内容能融进章节正文

数据库迁移此版加了 253 行，schema 大改。版本号到了 v0.1.0-beta.12。

几个值得说的设计

1. 章节正文不进 SQLite

ChapterRecord 只存 filePath 字段，正文落在项目目录的 .md 文件里。好处：

  • 用户可以脱离应用直接用 VS Code / Typora 改章节
  • Git 化项目目录就能版本控制
  • SQLite 文件保持小，不存大文本

读写统一走 packages/storage/src/fs-layout.ts 的 readChapterFile/writeChapterFile/nextChapterFileName。

2. IPC 走 TypeScript declaration merging

packages/shared/src/ipc.ts 里 IpcRequestMap interface 在不同位置多次声明，TS 自动合并。新增 IPC channel 只要在对应模块文件里 declare module 一下，不用动总入口 —— 大型 monorepo 的福音。

3. 4-Agent 自动写作管线

packages/auto-writer-engine 用 4 个 Agent 接力写章节：

  • Planner：根据大纲拆当前章节要点
  • Researcher：从 RAG / 世界书拉相关材料
  • Writer：实际产出正文
  • Reviewer：审稿打分，不达标退回 Writer 重写

Agent 之间走 streaming，通过 BrowserWindow.webContents.send 把 chunk 实时推到 renderer。

4. Skill Engine = 用户可编程的 Prompt 模板

packages/skill-engine 实现了一套类 Jinja 的模板语言，用户可以写自定义 Skill：调用变量、循环世界条目、嵌套调用其他 Skill。配合 world-info-activator 的智能激活，能根据当前章节自动决定哪些设定条目应当注入到 prompt 上下文。

踩过的坑（防止后来者再踩）

  1. shared / storage 必须先 build dist 再 typecheck desktop —— 它们以 workspace package 的 dist 形式被消费，不是 src
  2. migrations 永远 append-only —— 当前到 v18，永远不要改老版本，只加新对象
  3. @xyflow/react + @dagrejs/dagre 用于世界书关系图，react-flow 风格的 style import 容易踩
  4. better-sqlite3 prebuild 别用 npm rebuild —— 走 prebuild-install --runtime=electron --target=32.x.x 显式指定
  5. IPC channel 命名 kebab-case domain:action —— 强约定，全局搜索友好

下一步

beta.12 之后想推的方向：

  • 桌宠系统：右下角角色陪写，根据写作状态切换动画
  • 章节快照对比：每次保存自动留快照，支持时间轴回滚
  • 多语言 prompt 模板：当前 prompt 偏中文友好，国际化要补
  • 离线 LLM 支持：接 llama.cpp / ollama，真正做到完全离线

写在最后

30 天 30 commit 不算高频，但每个 commit 改动量都大 —— 这其实是不好的实践，下版本会用 Conventional Commits 强制拆分。

整个项目的设计哲学是 “用户的数据是用户的”：本地优先、文件可读、provider 自选、不绑定云。希望能成为认真写小说的人手里趁手的工具。

如果你在做类似项目或有兴趣聊聊，欢迎在 GitHub Issues 找我。
