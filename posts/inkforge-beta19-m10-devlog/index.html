InkForge 进度同步：从 beta.12 到 beta.19，先把写作链路打稳

发表于2026-06-19|更新于2026-06-19|项目记录
|总字数:3.5k|阅读时长:11分钟|浏览量:

上两篇《InkForge 开发实录》和《InkForge 打磨记》停在 beta.12。那时 InkForge 已经像一个完整工具了，但还没有真正经受”拿来写长篇”的压力测试。最近这一段主要不是堆功能，而是把最容易让作者失去信任的地方逐个补上：重复章节、截断正文、保存后旧内容回灌、IPC 边界、打包验证，以及一大批小而烦人的 UI 反馈问题。

如果说 beta.12 是”功能闭环”，那 beta.19 更接近”可以开始认真用”。

0. 这一轮先看结果

方向 更新
当前版本 v0.1.0-beta.19
写作链路 修复章纲重复、正文截断、重写后旧正文、未绑定章节；AI 写作页正式承接正文生成
工程安全 IPC 入参运行时校验、外链打开集中到主进程、终端生产环境门禁、更多主进程/service 测试
发布验证 Windows unpacked / setup / portable 验证，GitHub Actions 覆盖 verify、desktop e2e、packaged UI smoke
数据安全 新增项目备份包导出/导入设计，不导出密钥和全局模型配置
体验打磨 统一 loading 动效、短状态反馈、应用内确认、错误保留、reduced motion
世界观工具 世界图谱从”节点连线”升级到社群、关键连接点、孤立资料、关系质量和成熟度分析

这篇不再重复 monorepo、数据库表、AutoWriter 4-Agent 这些基础结构，前两篇已经写过了。这里主要记”从 beta.12 之后，哪些真实问题被修掉了”。

1. 最大的一条线：章纲到正文生成终于收住了

这几天最值得记录的不是某个按钮，而是产品职责被重新理顺了：

大纲页负责规划、选择、准备章节；AI 写作页负责正文生成、续写和精修。

之前大纲页里有一些”顺手生成正文”的路径，看似方便，实际上会让职责变得混乱：一个页面负责规划，另一个页面负责写作，中间再插一个小弹窗去生成和采纳正文，最后 bug 很难定位。现在这条链路被收回到 AI 写作页，旧的直出组件也开始清理。

重复章节

真实使用里最明显的问题是：大纲拆分完章节，再生成正文时会冒出重复章节。

根因不是”AI 多写了”，而是程序侧的绑定语义不够严格：同一张大纲卡已经对应某个章节时，再次提交草稿不能默认新建章节。现在 commitDraft 的语义被收紧：

  • 有明确 chapterId 时，覆盖目标章节。
  • 大纲卡已经绑定章节时，后续提交复用这个章节。
  • 没有目标的批量重写不偷偷新建章节。

这类 bug 很讨厌，因为单看 UI 像是”模型乱写”，实际是本地状态机在制造重复数据。

截断续写

另一个高频问题是长章写到一半突然停了。这里也分两层：

  1. 生成预算确实偏保守，长章容易碰到上限。
  2. provider 的 finish reason 如果在流结束时被 [DONE] 之类的事件覆盖，程序就看不出”这是被长度截断”。

现在处理方式更务实：提高章节生成预算，同时保留 provider 的真实完成原因；如果是 token limit，或者文本尾部明显像没收住，就触发补写，且补写 prompt 明确要求”只补后续，不重复前文，不重写标题”。

全部重写正文

重写正文也补齐了：

  • 已写章节可以单章重写。
  • 大纲页可以批量”重写全部正文”。
  • 重写前保留快照，避免覆盖后没有回退点。
  • 批量重写只处理已经有目标章节的大纲卡，不把未写卡片误当成”创建新章节”。

这个功能看起来只是一个按钮，但它背后需要 outline card -> chapter -> snapshot -> commitDraft -> cache invalidation 一整条链路都对齐。

2. 从”能跑”到”敢发 beta”：验证和打包开始像样了

beta.12 之前，项目能跑，但”能不能放心发给别人”还差一截。最近补的更多是工程底座。

IPC 从类型约束走向运行时校验

Electron 应用最危险的边界之一就是 renderer 到 main 的 IPC。TypeScript 能保证开发时类型，但运行时 payload 还是可能乱传。

现在主进程入口开始统一做 runtime parse：文件路径、二进制导入导出、模型调用、settings 写入、章节草稿提交、世界观和素材写入、provider/model 配置、enum/union 字段，都不应该直接相信 renderer 传来的对象。

并且加了一个很实用的回归测试：扫描 IPC handler，防止以后又把参数写回具体 *Input 类型，绕过运行时解析。

这不是用户能一眼看到的功能，但它直接影响本地写作数据的安全边界。

外链和终端边界

一些小安全债也顺手收了：

  • 外链打开集中到主进程，只允许 http/https，再交给系统浏览器。
  • HelpMenu 不再直接 window.open。
  • 内嵌终端在生产环境有主进程门禁，不只是前端隐藏按钮。
  • contextIsolation: true、nodeIntegration: false、sandbox: true 这些 Electron 基础安全状态重新确认过。

终端功能本身很有用，但它不该成为默认安装包的风险入口。当前策略是：开发场景可用，生产 beta 阶段先收住。

项目备份包

最近还补了一条很关键的”数据自治”能力：项目备份包导出/导入。

导出的是项目自己的内容：

  • 项目元数据
  • 章节 Markdown
  • 人物
  • 世界观
  • 素材
  • 参考库
  • 关系
  • 封面
  • 版本备份
  • 日志
  • 审查记录

不导出的是全局或敏感内容：

  • 模型服务密钥
  • provider/provider_keys
  • app settings
  • scene bindings
  • 工作区配置
  • CI/output 验证结果

导入时创建新项目，不覆盖原项目。ZIP 也加了路径安全校验：拒绝绝对路径、反斜杠、..、重复 entry、过大 entry 和过多 entry。

这件事和 InkForge 的核心价值观一致：用户的数据是用户的，而且应该能带走。

3. 真实验证不再只靠”我觉得可以”

这一轮很大一个变化是：验证开始覆盖真实路径。

本地和 CI 里陆续跑过这些：

  • pnpm typecheck
  • pnpm test
  • pnpm build
  • Electron e2e
  • Windows packaged UI smoke
  • Windows unpacked exe 启动冒烟
  • setup / portable 产物存在性、大小和 hash 检查
  • mock LLM 走真实 preload API、真实 IPC、真实 main 进程模型运行时
  • 真实模型短样例和长篇连续闭环 proof

这里最重要的是 mock LLM 那条：它不是绕过主进程直接测纯函数，而是让 AutoWriter / Review 真的走 Electron preload、IPC、main service、LLM runtime，只是把模型响应换成确定性 mock。这样没有 API key 也能在 CI 或本地稳定复现核心链路。

真实模型也跑过一轮长篇连续闭环：多章节、Planner / Writer / Critic / Reflector 都走完。它证明的不是”文笔一定好”，而是程序链路能撑住连续写作。

4. M10：不急着加功能，先让它像成熟软件

beta.19 后进入 M10，主题很明确：polish + ecosystem。

最近本地工作区里有一大批还没提交的 UI/UX 打磨，方向不是”更炫”，而是”别让用户怀疑软件坏了”。

统一加载动效

以前很多地方直接用 Tailwind 的 animate-spin、animate-pulse、animate-bounce。问题是：

  • 每个组件写法不一致；
  • reduced motion 下仍然可能无限转；
  • 加载图标切换会造成按钮宽度抖动；
  • 有些状态只有文字，没有明确的视觉反馈。

现在新增了统一的 MotionSpinner / MotionSpin，等待态、刷新态逐步迁过去。自动写作、资料检索、审稿、世界包、角色导入、酒馆、声音档案、书架、素材页这些高频路径都在收口。

短状态反馈生命周期

另一个隐蔽但高频的问题是：成功提示 2 秒后消失，通常会写一个 setTimeout。页面一多，这些 timer 就到处都是；组件卸载、弹窗重开、连续点击时，旧 timer 很容易把新状态清掉。

所以抽了一个 useTimedStatus()，统一做：

  • 显示新状态前清掉旧 timer；
  • 组件卸载时清理；
  • 成功/取消可以自动消失；
  • 错误保留，不要用户还没看清就没了；
  • 需要读屏提示时用 role="status" / role="alert"。

这条线改了很多组件：设置、模型服务、素材、世界包、审稿、快照、每日总结、桌宠、来信、成就、终端、帮助菜单、标题栏等。单个改动都不大，但加起来应用会稳很多。

原生弹窗替换成应用内确认

浏览器原生 confirm() 在桌面应用里体验很割裂，甚至可能因为焦点/遮挡看起来像”按钮没反应”。最近开始把删除、清空、危险操作改成应用内确认：按钮进入确认态、错误有明确反馈、操作中禁用，避免重复触发。

这类细节很小，但对写作软件很重要。作者正在写东西时，不应该被突兀的浏览器弹窗打断。

5. 世界图谱变得更像真正的设定工具

世界观关系图是这轮最有”新鲜感”的地方。

以前它更像一个节点连线图：人物、世界条目、关系，能看，但信息密度有限。现在它开始像一个结构分析工具：

  • 搜索
  • 人物 / 世界条目筛选
  • 世界条目分类筛选
  • 最低关系强度
  • 只看已连接资料
  • 全图 / 一跳 / 二跳焦点范围
  • 分层布局 / 放射布局
  • MiniMap
  • 最短关系路径
  • 关系社群
  • 关键连接点
  • 未接入资料
  • 关系质量问题
  • 图谱成熟度评分
  • 悬停邻域高亮

更重要的是，图算法被拆到了 world-graph-logic.ts，并补了测试。这样图谱不是一坨 TSX 里塞满业务逻辑，而是：

  • 纯逻辑：路径、社群、桥接点、孤立点、成熟度评分；
  • UI：展示、筛选、交互、动效。

还有一个很实用的小点：图谱视图状态按项目记忆。搜索内容、布局模式、焦点范围、筛选、最低关系强度这些视图偏好可以保存在 localStorage。注意只存”怎么看图”，不存世界观正文。

我挺喜欢这个方向。小说世界观不是简单资料库，关系图如果能主动告诉作者”哪些设定没接上”、”哪些人物是关键桥接点”、”哪些关系强度太弱”，它就从展示工具变成了整理工具。

6. 修了一个最危险的错觉：输入后消失，刷新后又出现

这个问题值得单独写，因为它会极大伤害用户信任。

现象是：输入文字后内容消失，但刷新后又出现。乍看像”没保存”，实际更糟也更隐蔽：内容已经保存了，但旧数据回流把本地输入覆盖掉了。

这类问题通常出现在三个地方：

  1. 编辑器内部状态；
  2. React state；
  3. 查询缓存 / 父级刷新回来的旧数据。

最近修了几道防线：

  • NovelEditor 跳过自己刚发出的 onChange 回流，避免受控组件反向 setContent 清掉正在输入的字；
  • IME 中文输入合成中不 setContent；
  • EditorPane 用 setEditorContent 同步 React state、contentRef 和本地章节缓存；
  • 保存成功、快照恢复后同步更新 chapter-content 查询缓存，避免章节列表刷新后拿旧正文回灌；
  • WorldEntryDetail 用 loaded entry id、dirty snapshot、updatedAt 做防线，同一条目刷新时不覆盖正在输入的内容。

写作软件里，”字会不会丢”是底线。这个问题的修复优先级比很多新功能都高。

7. 当前仍然没做完的事

透明一点说，beta.19 不是”完全完成”。

本地 M10 打磨还没完全提交

现在本地还有一大批 UI polish WIP。它们已经跑过多轮 typecheck / build / test / a11y / diff check，但还需要按主题拆 commit：动效统一、状态反馈、世界图谱、输入同步、World Pack 融合保存等，不适合一把梭。

Windows 仍未签名

0.1.x 阶段构建还是未签名。Windows SmartScreen 可能提示未知发布者。对 beta 可以接受，但正式分发前需要代码签名证书。

测试还可以继续扩

现在测试比之前扎实很多，但仍然能继续补：

  • storage repository 更完整的 migration / 默认值 / parse 测试；
  • provider/registry；
  • research adapter fallback；
  • chapter-service、world-pack-service；
  • 完整写作链路 e2e。

世界图谱还可以继续长

现在图谱已经有结构洞察，但还可以继续补图例、关系编辑的批量操作、导出结构摘要、和 AI 写作上下文联动。

写在最后

这一轮最大的感受是：功能闭环之后，真正耗时间的是”让它别在真实使用里背刺用户”。

重复章节、截断正文、保存后旧内容回灌、按钮点了没反馈、错误提示一闪而过、加载状态抖动、原生弹窗打断上下文、打包后 native 模块丢失 —— 这些都不是宣传页上会写的功能，但它们决定一个工具能不能被长期信任。

InkForge 的方向还是那句：本地优先，数据是用户的，AI 是协作者，不是云端黑盒。

beta.12 证明它”有完整形状”。beta.19 之后，我更关心的是：它能不能陪一个作者稳定写下去。
